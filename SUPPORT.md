# Support

This project uses a public, issue-driven support model in v1.

## Start Here

Before opening an issue:

- Read the [README](README.md) for setup and architecture context.
- Check the relevant docs under [`docs/`](docs/).
- Search existing issues and pull requests for the same question or problem.

## Which Issue Form To Use

- **Bug report**: Reproducible defects, incorrect behavior, crashes, or broken docs.
- **Question**: Usage help, configuration questions, or "am I doing this right?" requests.
- **Feature request**: Improvements to the product, docs, or contributor experience.

Security vulnerabilities must not be filed as public issues. Follow
[SECURITY.md](SECURITY.md) instead.

## What To Include

Include the details that make the issue actionable:

- What you were trying to do
- What you expected to happen
- What happened instead
- Reproduction steps, commands, logs, screenshots, or sample payloads when relevant
- Your environment details, including OS, Python version, and any local tooling involved

## Support Model

- Support happens through GitHub issues only in v1.
- Support is best effort and provided without an SLA.
- There is no private support inbox, guaranteed response time, or paid support path in this repository.
